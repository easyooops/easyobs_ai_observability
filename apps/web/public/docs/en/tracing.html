Tracing — Full Feature Manual

The Tracing page is the primary interface for exploring every agent invocation captured by EasyObs. It provides a dual-view (Traces / Spans), rich filtering, configurable columns, a slide-out inspector drawer, and a graph-based flow visualizer.

Page Layout

The Tracing page is divided into three regions:

RegionPositionPurpose
Filter PanelLeft sidebarStatus, Session, Service, and Column toggle filters
Data TableCenterPaginated list of Traces or Spans with sortable columns
Inspector DrawerRight panel (on row click)Detailed view of the selected trace with 4 sub-tabs

Tabs: Traces vs. Spans

The top segmented control switches between two perspectives:

Traces Tab (default)

Shows one row per complete agent invocation (root span + all children). Each row shows:

ColumnDefault VisibleDescription
StartedYesTimestamp (absolute + relative "3m ago" format)
OperationYesRoot span name (e.g. "agent.turn")
ServiceYesService name tag from service.name resource attribute
DurationYesEnd-to-end latency with a proportional mini-bar (max 2s scale)
SpansYesTotal span count within the trace
StatusYesOK (green) / ERROR (red) / UNSET (gray)
TokensNo (LLM mode)Sum of o.tok.in + o.tok.out across all LLM spans
CostNo (LLM mode)USD cost (o.price) formatted as $0.0012
ModelNo (LLM mode)Primary LLM model name used in the trace
SessionNo (LLM mode)Session ID (o.sess) for multi-turn grouping
Trace IDNoFull 32-char hex trace identifier
LLM Column Auto-Switch: When arriving from the Sessions page via ?session=..., the Tokens/Cost/Model/Session columns are automatically toggled on so you immediately see LLM-relevant data.

Spans Tab

Shows individual spans (flattened, one row per span). Columns:

ColumnDescription
NameSpan operation name
Kindo.kind attribute badge: agent / llm / retrieve / tool / chain
ModelLLM model name (if kind=llm)
ServiceService name
DurationSpan-level latency with mini-bar
TokensPer-span token count
CostPer-span USD cost
StatusOK / ERROR / UNSET
TraceParent trace ID (truncated, clickable → detail)

Filter Panel (Left Sidebar)

Status Filter

Multi-select checkboxes for OK, ERROR, UNSET. Each shows the count of matching traces. Click "reset" to clear.

Session Filter

Exact-match text input for o.sess. When set, only traces tagged with that session ID are shown. A chip appears in the page header showing the active filter with a × clear button.

Service Filter

Multi-select checkboxes showing up to 10 services, ordered by trace count (descending). Each shows the count.

Column Toggles

Checkboxes for all 11 columns. Toggle any column on/off. Click "reset" to return to the default set (first 6 visible).

Pagination

  • Rows per page: Select from 25 / 50 / 100 / 200
  • Page navigation: ← Previous / Current/Total / Next →
  • Page resets to 1 when filters or time window change

Global Search Integration

The top-bar search field (keyboard shortcut: /) filters traces/spans in real-time by matching against: root name, trace ID, service name (case-insensitive substring match).

Time Window Integration

The top-bar time window (1h / 6h / 24h / 7d / Custom) determines which traces are fetched. With Live mode active, the table auto-refreshes every 5 seconds.

Trace Inspector Drawer

Click any row to open the right-side inspector drawer. The drawer has a header with action buttons and 4 sub-tabs:

Drawer Header Actions

ButtonAction
Evaluate →Navigate to Quality > Runs with this trace pre-filled for evaluation
+ LabelNavigate to Golden Sets > Human Labels with this trace ID pre-filled
Full view →Open Full View: dedicated trace page with LLM summary, span tree, flow DAG, timeline waterfall, and per-span inspector
×Close the drawer

Tab: Summary

Key-value overview of the trace:

  • Operation → Root span name
  • Service → Service name
  • Status → Color-coded badge
  • Duration → End-to-end latency
  • Spans → Total count (N error)
  • Session → Session ID (if present)
  • Tokens → Input / Output token breakdown
  • Cost → Total USD cost
  • Models → All LLM models used (as chips)
  • Started → Absolute timestamp
  • Trace ID → Full identifier

Below the KV list:

  • View Span Flow → Opens a centered modal with a directed graph of the trace (one node per span, parent/child edges, duration labels; colors follow span kind — see Supported Span Kinds). The inspector screenshot above includes this modal so a separate figure is not repeated here.
  • First spans → Text preview of the first spans with indentation, names, and durations

Tab: LLM

Formatted LLM summary card showing:

  • Model name and provider
  • Input prompt (formatted, scrollable)
  • Output completion
  • Token counts (in/out)
  • Cost breakdown
  • Latency

Tab: Attributes

Full key-value list of all OpenTelemetry attributes on the root span. Includes both standard attributes and EasyObs o.* namespace attributes.

Tab: Events

List of all span events across all spans in the trace. Each event shows its name and the span it belongs to. Useful for seeing exceptions, log messages, or custom events recorded during execution.

Full View (trace detail page)

Full View is the dedicated, full-screen trace experience for a single trace. Open it from the inspector drawer via Full view →, or navigate directly to /workspace/tracing/detail/?id=<traceId>. The page is built from the same NDJSON span payload as the list view but arranges analysis into a header, an LLM-centric summary band, and a four-column workspace below.

Header strip

  • Operation — Root span name (e.g. agent.rag.plan) with an OK / error badge
  • Context line — Short note that the view is a waterfall derived from stored spans
  • Trace ID — Identifier (often truncated in the UI with full value available in copy/inspector)
  • Environment / service — e.g. service or env label such as demo
  • Total duration — End-to-end latency for the whole trace
  • Evaluate — Jump to Quality workflows with this trace in context
  • + Human label — Jump to human labeling / golden workflows for this trace

LLM summary band

A horizontal summary aggregates what matters for LLM agent traces before you drill into spans:

  • Session & identity — Session id, user (if present), request id
  • Shape of work — Counts such as how many LLM, retrieve, and tool spans appear in the trace
  • Usage & cost — Token totals (in/out), estimated USD cost, primary model name
  • Conversation preview — First captured user input and last captured assistant output as quick sanity checks

Four-panel workspace

The lower area stays in sync: selecting a span in one panel updates the others where applicable.

PanelRole
Span treeHierarchical list of all spans (root and children) with durations. Fast navigation and mental model of nesting.
FlowVisual DAG: parent node at the top, children below, edges showing call structure (same tree as the modal graph, laid out for the detail page).
TimelineWaterfall / Gantt-style chart over the trace clock: bar length = duration, vertical stacking = nesting, so overlap and ordering are obvious.
InspectorDetail for the selected span: tabs such as LLM (prompt/response when applicable), Attributes, Events (with counts), and Raw. Shows span id, start time, and other fields needed for debugging.

Use Full View when the drawer is too narrow: long prompts, deep trees, comparing timeline vs graph, or stepping span-by-span with the inspector while keeping the waterfall visible.

Cost & Token Tracking

The pricing pipeline works as follows:

  1. Agent SDK records o.tok.in and o.tok.out on each LLM span
  2. On ingest, the server enriches with o.price = tokens × per-token rate
  3. Pricing source priority: tokencost → litellm → built-in fallback table
  4. Configurable via EASYOBS_PRICING_SOURCE
  5. Cost is summed across all LLM spans for the trace-level total

Data Flow

UI/UX perspective: what operators feel as “near real-time trace visibility” is this ingest pipeline from SDK export to parsed, enriched, queryable rows.
