LLM 저지 — 전체 기능 매뉴얼

LLM 저지(Judge)는 자동 평가자 역할을 하도록 EasyObs에 등록된 외부 언어 모델 엔드포인트입니다. LLM의 입력/출력을 분석해 verdict와 함께 근거(reasoning)를 산출합니다.

개념

LLM 저지는 다음과 같이 동작하도록 설정된 모델 엔드포인트입니다.

  1. 구조화된 평가 프롬프트(트레이스 컨텍스트 + 평가 기준)를 입력으로 받는다.
  2. verdict (PASS / WARN / FAIL), 0.0~1.0 점수, 근거 텍스트를 반환한다.
  3. 합의 정책을 통해 다른 저지와 결합되어 더 신뢰성 있는 결과를 만들 수 있다.

다중 저지 합의

평가 프로파일이 여러 저지를 사용할 경우, verdict는 프로파일별로 설정된 합의 정책에 따라 결합됩니다.

정책판정 로직활용 예시
Single첫 번째 저지의 verdict를 최종 결과로 사용속도와 비용을 우선시하는 평가
Majority저지의 50% 초과가 동의해야 verdict 확정신뢰성과 비용의 균형
Unanimous모두 동의해야 하며, 그렇지 않으면 WARN의심이 있으면 무조건 플래그하는 고위험 의사결정
Weighted각 저지의 weight로 가중 평균전문 영역 저지에 가중치 3배 등 앙상블 운영

가중치 합의 예시

예: 3개의 저지가 가중치 1.0, 1.0, 3.0 (합 = 5.0)을 가질 때

  • Judge A (w=1.0): PASS → PASS에 1.0 기여
  • Judge B (w=1.0): FAIL → FAIL에 1.0 기여
  • Judge C (w=3.0): PASS → PASS에 3.0 기여
  • 결과: PASS 점수 = 4.0/5.0 = 0.80 → verdict PASS

편집 & 삭제

  • Edit: 행 클릭 → 현재 값이 채워진 폼 → 수정 → Save
  • Delete: 휴지통 아이콘 클릭 → 확인 다이얼로그 → 영구 삭제
  • 이미 활성 프로파일이 참조 중인 저지를 삭제해도 기존 실행은 깨지지 않지만, 새 실행에서는 해당 저지를 사용할 수 없습니다.

모범 사례

  • 교차 검증을 위해 서로 다른 provider의 저지를 최소 2개 등록하세요.
  • 최대한 재현 가능한 결과를 위해 temperature: 0을 사용하세요.
  • 가격 값을 정확히 설정해야 Budget Guard 계산이 정확해집니다.
  • 저지 이름은 의도가 드러나도록 작성하세요 (예: "GPT-4o Strict Safety" vs "Claude Helpfulness").
  • 실험 중인 저지는 비활성화해 프로덕션 실행에서 제외하세요.

Evaluation Prompts (버전 관리되는 프롬프트)

Judge Models 옆의 Evaluation Prompts 탭에서는, LLM-as-a-Judge가 평가 차원별로 사용하는 system 메시지와 user 메시지 프롬프트를 조직 단위로 커스터마이즈할 수 있습니다.

개념

각 평가 차원(Faithfulness, Answer Relevance, Coherence 등)은 다음 두 프롬프트 한 쌍을 가집니다.

  • System Prompt — 저지 LLM에게 평가 방식을 안내하는 지침
  • User Message Template — {rubric_id}, {rubric}, {context_json} 같은 placeholder를 포함하는 구조화된 페이로드

프롬프트는 버전 관리됩니다: v1(초기/기본) → v2 → v3 …. 편집할 때마다 새 버전이 생성되며, 평가 시점에는 active 버전만 사용됩니다. 이전 버전은 감사와 롤백을 위해 보존됩니다.

지원 평가 차원

ID이름설명
faithfulness충실도응답이 제공된 컨텍스트와 사실적으로 일치하며 환각 없이 작성되었는지 평가합니다.
answer_relevance답변 관련성응답이 사용자의 질문을 얼마나 직접적이고 완전하게 다루는지 평가합니다.
context_utilization컨텍스트 활용도모델이 검색된 컨텍스트를 답변 구성에 얼마나 효과적으로 활용했는지 평가합니다.
context_recall컨텍스트 재현율그라운드 트루스에 포함된 관련 정보가 검색된 컨텍스트에 모두 담겨 있는지 평가합니다.
context_precision컨텍스트 정밀도검색된 컨텍스트가 주제에 집중되어 있고 무관한 정보가 섞이지 않았는지 평가합니다.
answer_correctness답변 정확성응답의 사실 정확도를 알려진 그라운드 트루스 답변과 비교해 검증합니다.
harmfulness_safety안전성 / 유해성생성된 응답에 포함된 유해·악성·위험 콘텐츠를 탐지합니다.
noise_sensitivity노이즈 민감도입력 컨텍스트에 무관하거나 노이즈성 정보가 섞였을 때의 견고성을 평가합니다.
coherence일관성응답의 논리적 흐름, 일관성, 구조적 명료함을 평가합니다.
answer_conciseness간결성응답이 불필요한 장황함이나 반복 없이 간결하게 작성되었는지 평가합니다.
tool_use_quality도구 사용 품질tool / function 호출의 선택, 파라미터 사용, 결과 해석의 정확성을 평가합니다.

사용 방법

  1. Quality → Judges → Evaluation Prompts 탭으로 이동
  2. 좌측 패널에서 차원(dimension) 선택
  3. 기존 버전 확인 (커스텀 프롬프트가 없으면 "default"로 표시)
  4. "New version"을 클릭해 커스텀 프롬프트 생성
  5. System Prompt 또는 User Message Template 편집
  6. "Save as new version" 클릭

버전 관리

액션동작
첫 프롬프트 생성v1로 저장되며 자동으로 active로 설정됩니다.
편집 / 새 버전이전 버전이 비활성화되고 새 버전이 active가 됩니다.
이전 버전 활성화active 플래그를 전환합니다 (삭제 없이 롤백).
Clone & Edit기존 버전 내용을 채운 상태로 편집기를 엽니다.

프롬프트가 평가에 적용되는 우선순위

저지 LLM 호출 시 프롬프트 적용 우선순위는 다음과 같습니다.

  1. 프로파일 단위 오버라이드 — 프로파일에 judge_system_prompt 또는 judge_user_message_template이 설정되어 있으면 가장 우선합니다.
  2. DB의 버전 관리되는 프롬프트(active) — 프로파일 오버라이드가 없으면 해당 차원의 active 버전이 사용됩니다.
  3. 내장 기본값 — 위 두 가지가 모두 없으면 하드코딩된 기본값으로 폴백됩니다.
