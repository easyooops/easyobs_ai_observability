골든 셋 — 전체 기능 매뉴얼

골든 셋은 회귀 테스트와 저지 캘리브레이션에 사용되는 큐레이션된 그라운드 트루스 데이터셋입니다. 사람이 검증한 입력/출력 쌍과 기대 verdict를 포함하여 재현 가능한 품질 벤치마크를 가능하게 합니다.

개념

골든 셋이 답하는 질문: "이 정확한 입력에 대해 올바른 기대 동작은 무엇인가?"

  • 프롬프트, 모델, 아키텍처가 변경될 때 품질 회귀를 탐지합니다.
  • LLM 저지의 정확도를 보정하기 위한 그라운드 트루스로 활용됩니다.
  • CI/CD 품질 게이트를 가능하게 합니다 (golden set 실행 → 모두 pass → 배포).

골든 셋을 만드는 3가지 진입점

#방법출처설명
1Manual UploadCSV / JSON 파일기대 출력과 verdict가 미리 준비된 테스트 케이스를 업로드합니다.
2Auto-SynthesizedLLM 생성기존 트레이스 패턴으로부터 LLM이 테스트 케이스를 자동으로 생성합니다.
3Trace Labeling실시간 트레이스실제 운영 트레이스를 사람이 그라운드 트루스로 라벨링합니다.

진입점 1: Manual Upload

CSV 형식

input,expected_output,verdict,notes
"What is Python?","Python is a programming language...","pass","Basic knowledge"
"Write harmful content","I cannot help with that.","pass","Safety check"
"2+2=?","5","fail","Math accuracy — wrong answer"

JSON 형식

[
  {
    "input": "What is Python?",
    "expected_output": "Python is a programming language...",
    "verdict": "pass",
    "dimensions": {
      "accuracy": 1.0,
      "helpfulness": 0.9
    },
    "notes": "Basic knowledge"
  }
]

업로드 절차

  1. Quality → Golden Sets로 이동합니다.
  2. "Create Golden Set"을 클릭합니다.
  3. 이름과 설명을 입력합니다.
  4. "Upload File"을 선택합니다.
  5. CSV/JSON 파일을 드래그하거나 선택합니다.
  6. 미리보기 표에서 파싱된 항목을 확인합니다.
  7. Save를 클릭합니다.

진입점 2: Auto-Synthesized

합성기는 기존 트레이스 패턴을 분석하여 테스트 케이스를 생성합니다.

  1. 기존 서비스와 시간 윈도우를 선택합니다.
  2. 시스템이 대표 트레이스를 샘플링합니다.
  3. LLM이 변형 케이스(엣지 케이스, adversarial 입력, 경계 조건)를 생성합니다.
  4. 생성된 각 케이스에는 기대 출력과 verdict가 포함됩니다.
  5. 사람이 검토하고 항목을 승인 또는 거절합니다.
참고: 자동 합성은 등록된 저지 모델을 사용하므로 API 비용이 발생합니다. 확정 전에 비용 예측치를 확인하세요.

진입점 3: Trace Labeling

실제 운영 트레이스를 골든 셋 항목으로 변환합니다.

  1. Tracing 페이지에서 임의의 트레이스에 있는 "+ Label"을 클릭합니다.
  2. 해당 트레이스의 입력/출력이 캡처됩니다.
  3. 리뷰어가 verdict(Pass / Warn / Fail)를 부여합니다.
  4. 필요 시 디멘션 점수와 notes를 추가합니다.
  5. 라벨링된 트레이스는 Human Labels 풀에 누적됩니다.
  6. 선택한 휴먼 라벨로부터 골든 셋을 생성합니다.

골든 셋 구조

필드설명
ID자동 생성되는 고유 식별자.
Name사람이 읽기 쉬운 이름 (예: "Safety Regression v3").
Description셋의 목적과 컨텍스트.
Entries테스트 케이스 배열 (아래 항목 구조 참고).
Source생성 방식 (manual / synthesized / labeled).
Created생성 timestamp.
Last Run이 셋이 회귀 실행에 가장 최근에 사용된 시각.
Last Pass Rate가장 최근 실행의 결과.

항목 구조

필드필수 여부설명
inputYes테스트할 사용자 입력 / 질의.
expected_outputNo비교 기준이 되는 "정답" 출력.
verdictYes기대 verdict: pass / warn / fail.
dimensionsNo디멘션별 기대 점수 (0.0~1.0).
notesNo해당 verdict가 기대되는 이유에 대한 컨텍스트.
trace_idNo(라벨링으로 만든 경우) 원본 트레이스 ID.

골든 셋 상세 — 사이드 드로어

Quality → Golden Sets에서 표의 임의 행을 클릭하면 우측 드로어(모달 패널)가 열립니다. 어두워진 배경을 클릭하거나 Escape를 누르면 닫힙니다. 셋을 선택하면 내부 탭은 Items로 초기화됩니다.

헤더 스트립 — 3개 패널 요약

드로어 상단의 요약 카드는 셋 이름, mode(regression / cohort / synthesized), layer(L1 / L2 / L3), 항목 개수, revision 개수, Agent 엔드포인트 설정 여부, 생성 시각을 표시합니다. 그 아래 3개 컬럼이 다음을 요약합니다.

패널내용
Datalayer별(L1 / L2 / L3) 항목 개수, 항목 source(manual, upload, trace label, auto-gen 등) 분포, review 상태(unreviewed / reviewed / disputed).
Trust평가 실행 후 inter-rater 메트릭(예: Cohen κ, Fleiss κ, Krippendorff α, multi-judge agreement, human-judge κ)과 임계값 대비 pass/fail 색상, rater / judge / disputed 카운트.
Usage이 셋과 연결된 평가 실행, 마지막 pass rate, 최근 실행들의 단순 pass rate 추세.

내부 탭 (6개)

탭목적
Items골든 항목을 탐색·관리합니다. 추가 방법: Manual(JSON payload), From trace(trace id를 붙여 query/response 스냅샷), Auto-discover(셋이 서비스에 연결된 경우 해당 서비스의 최근 트레이스 샘플링). 권한 범위 내에서 review 상태 수정, 항목 삭제 등을 수행할 수 있습니다.
Regression Run평가 프로파일과 옵션 revision을 선택해 저장된 Agent 엔드포인트로 Golden Regression Run을 시작합니다. Agent 연결이 필요합니다. SSE로 phase/progress가 실시간 업데이트되며, 진행 중 run id는 session storage에 저장되어 페이지가 다시 연결할 수 있습니다. 권한이 있으면 진행 중인 실행을 취소할 수 있습니다.
+ Auto-generateLLM 기반 합성기로 이 셋의 새 항목을 제안합니다. API 호출 비용이 발생하며, 항목으로 승격하기 전에 결과를 검토해야 합니다.
+ Upload이 셋으로 CSV/JSON 대량 임포트하며, 미리보기/검증을 제공합니다.
Agent connectionPOST 엔드포인트 URL, JSON 요청 템플릿, Vault auth ref, timeout, max concurrency를 설정합니다. {{query_text}}, {{run_id}}, {{item_id}} 플레이스홀더는 실행 시 치환됩니다. Test connection으로 저장 없이 검증하고, Save로 설정을 영속화합니다. 유효한 엔드포인트가 저장되어야 회귀 실행을 시작할 수 있습니다.
Revisions / Trust좌측: revision 목록 (revision 번호, immutable vs mutable, 항목 수, 생성 시각). 우측: 선택된 revision에 대한 상세 Trust 메트릭 (Cohen/Fleiss/Krippendorff, multi-judge, human-judge, rater/judge/disputed 카운트). 평가 활동 이후에 채워집니다.

리스트 컬럼 (드로어 열기 전)

메인 표는 Name(설명 포함), Mode, Layer, Agent 설정 여부, Service, Items 개수, Created, Delete(권한이 있을 때) 컬럼을 표시합니다. New golden set으로 이름, mode(regression / cohort / synthesized), 설명, 자동 추천 또는 직접 지정한 layer, 옵션 service 스코프를 지정해 셋을 생성합니다.

Human Labels 서브 페이지

세그먼트 컨트롤로 ?tab=labels로 전환하면 (트레이스 어노테이션의) 휴먼 라벨 풀을 볼 수 있습니다. 셋별 드로어와는 분리되어 있지만 동일한 Golden Sets 영역 아래에 있습니다.

회귀 테스트 실행

골든 셋으로 에이전트가 여전히 허용 가능한 출력을 만드는지 확인합니다.

  1. Quality → Runs로 이동합니다.
  2. "golden_judge" 또는 "golden_gt" 모드의 프로파일을 선택합니다.
  3. 데이터 소스로 골든 셋을 선택합니다.
  4. 실행을 시작합니다.
  5. 시스템은 각 항목을 실시간 에이전트로 다시 실행하고, 결과를 기대 verdict와 비교합니다.
  6. 결과: 항목별 pass/fail과 diff 하이라이트.

휴먼 라벨 어노테이션

Human Labels 섹션(Quality → Golden Sets → Labels 탭)은 모든 트레이스 어노테이션을 보여줍니다.

컬럼설명
Trace ID원본 트레이스 (클릭 가능).
Verdict사람이 부여: Pass / Warn / Fail.
Annotator라벨러의 사용자명.
Created생성 timestamp.
Notes자유 텍스트 컨텍스트.

Best Practices

  • 통계적 유의성을 위해 골든 셋당 최소 50~100개 항목을 유지합니다.
  • 엣지 케이스(빈 입력, 매우 긴 입력, adversarial 프롬프트)를 포함합니다.
  • 골든 셋은 버전 관리합니다(v1, v2). 기존 버전을 수정하지 말고 새 버전을 만드세요.
  • 운영 배포 전마다 회귀 테스트를 실행합니다.
  • 에이전트의 기대 동작이 진화함에 따라 항목을 주기적으로 검토하고 업데이트합니다.
  • 합성기를 사용해 미처 고려하지 못한 엣지 케이스를 발굴합니다.

권한

  • SA / PO: 골든 셋 생성·업로드·수정·삭제 및 휴먼 라벨 추가
  • DV: 골든 셋 조회 및 본인 서비스 트레이스에 대한 휴먼 라벨 추가
