52개 평가 메트릭 — 전체 카탈로그

EasyObs는 Query → Retrieval → Context → Response → Agent/Tool → Operations → Human Feedback에 이르는 파이프라인 전 단계에 정렬된 메트릭 프레임워크로 에이전트 품질을 평가합니다. 각 메트릭은 "무엇이 잘못되었는가?"라는 구체적인 질문에 답하며, 실행 가능한 개선 카테고리와 매핑됩니다.

이 카탈로그를 읽는 방법

컬럼의미
Code짧은 식별자 (예: A1, B3, D4)
ModeR = Rule (결정적, 무료), J = LLM Judge (토큰 단위 과금), H = Human label
GT필요한 골든 셋 layer: L1(intent), L2(retrieval docs), L3(expected output). — 는 필요 없음
Cause Codefinding에 부착되어 Improvement Pack을 구동하는 구조화된 실패 코드
왜 52개인가? 에이전트는 파이프라인 단계마다 서로 다른 이유로 실패합니다. 단일한 "품질 점수"만으로는 무엇을 고쳐야 하는지 알 수 없습니다. EasyObs는 품질을 7개 그룹의 52개 메트릭으로 분해하여 정확히 어떤 단계에서 실패했는지 짚어내고, Improvement Pack을 통해 구체적인 개선 액션과 연결합니다.

Group A — Query Understanding (5개 메트릭)

목적: 처리가 시작되기 전에 에이전트가 사용자 의도를 정확히 이해했는지 평가합니다. 이 단계의 실패는 모든 후속 단계에 그대로 전이됩니다.

얻을 수 있는 인사이트: 의도를 잘못 분류하거나 질의 재작성이 어긋나거나 다국어/모호한 입력을 다루지 못한다면, 검색·생성을 튜닝하기 전에 이 영역부터 고쳐야 합니다.

CodeMetricModeGTCause Code설명 & 의의
A1Intent classification accuracyR/JL1query.intent_mismatch에이전트가 올바른 intent(예: "주문 상태" vs "반품")를 식별하는가? 분류 오류는 질의가 잘못된 워크플로로 라우팅되어 전체 실패로 이어집니다.
A2Query rewrite semantic preservationJ—query.rewrite_drift검색용으로 질의를 재작성/확장할 때 원래 의미가 보존되는가? drift가 발생하면 무관한 검색 결과로 이어집니다.
A3Ambiguity detection rateJ—query.ambiguous질의가 너무 모호해 추가 확인이 필요한 경우를 인지하는가? 놓치면 환각·임의 답변을 유발합니다.
A5Language detection accuracyR—query.lang_mismatch사용자가 사용한 언어로 응답하는가? 언어 불일치는 사용자 경험을 크게 해칩니다.
A6Query complexity scoreR—query.complexity_high에이전트의 처리 한계를 넘는 질의를 표시합니다. 휴먼 에이전트나 특화 모델로 라우팅할 근거로 활용됩니다.

Group B — Retrieval Quality (10개 메트릭)

목적: RAG 검색 단계에서 적절한 문서를 좋은 순위로 가져오는지 평가합니다. RAG 시스템에서 가장 자주 실패하는 영역이라 메트릭 수가 가장 많습니다.

얻을 수 있는 인사이트: 임베딩 모델, 청크 전략, reranker, 인덱스 커버리지, 지연 시간 중 정확히 어디를 개선해야 하는지 알 수 있습니다.

CodeMetricModeGTCause Code설명 & 의의
B1Recall@KRL2retrieval.recall_low관련 문서 중 상위 K개 안에 들어온 비율. recall이 낮으면 답변에 들어가야 할 핵심 정보가 빠집니다.
B2Precision@KRL2retrieval.noise_high상위 K개 중 실제로 관련된 문서 비율. precision이 낮으면 컨텍스트가 노이즈로 오염되어 LLM이 혼란스러워집니다.
B3MRR (Mean Reciprocal Rank)RL2retrieval.first_hit_late첫 번째 관련 문서가 얼마나 일찍 등장하는가? MRR이 낮으면 모델이 깊은 컨텍스트를 훑어야 유용한 정보를 찾게 됩니다.
B4HitRate@KRL2retrieval.miss이진 지표: 상위 K개 안에 관련 문서가 하나라도 있는가? "miss"이면 관련 컨텍스트가 전혀 없어 사실상 답이 망가집니다.
B5nDCG@KRL2retrieval.rank_qualityNormalized Discounted Cumulative Gain — 위치 가중치를 적용한 랭킹 품질. 관련 문서가 뒤로 밀릴수록 점수가 낮아집니다.
B6MAP (Mean Average Precision)RL2retrieval.precision_low모든 관련 문서에 대한 평균 precision. 관련 문서가 상단에 모이지 않고 흩어져 있으면 페널티가 부과됩니다.
B7Context precision (judge)J—retrieval.chunk_irrelevantLLM 저지가 평가: 검색된 청크가 정말 답변에 유용한가? 키워드 메트릭이 놓치는 의미적 무관성을 잡아냅니다.
B8Context recall (judge)JL3retrieval.coverage_lowLLM 저지가 평가: 검색된 컨텍스트가 기대 답변을 만들기에 충분한 정보를 담고 있는가? 지식 누락을 탐지합니다.
B10Reranker gainRL2retrieval.rerank_no_gainreranker가 원시 검색 대비 랭킹을 개선하는가? gain이 0이면 reranker가 지연만 늘리고 효용이 없으므로 제거를 검토하세요.
B11Retrieval diversityR—retrieval.dup_high검색된 청크가 서로 너무 비슷한가? 중복이 많으면 컨텍스트 윈도우를 낭비합니다.
B12Retrieval latency budgetR—retrieval.latency_over검색이 시간 예산 안에 끝나는가? 검색이 느리면 전체 응답이 지연되고 사용자 경험이 나빠집니다.

Group C — Context Quality (6개 메트릭)

목적: 검색은 끝났지만 생성이 시작되기 전 단계에서, 청크 분할·정렬·중복 제거 같은 컨텍스트 조립 품질을 평가합니다. 좋은 문서를 가져왔어도 조립이 나쁘면 LLM 출력이 떨어집니다.

얻을 수 있는 인사이트: 청크 크기, 정렬 전략, 중복 제거, 의미 유사도 필터링 등 컨텍스트 조립 파이프라인 중 어디를 개선해야 하는지 알 수 있습니다.

CodeMetricModeGTCause Code설명 & 의의
C1Chunk noise ratioR/J—context.noise컨텍스트 토큰 중 질의와 무관한 비율. 노이즈가 많으면 신호가 희석되어 LLM이 핵심을 놓치거나 환각을 일으킵니다.
C2Duplicate chunk ratioR—context.dup거의 중복인 청크 비율. 중복은 컨텍스트 윈도우를 낭비하면서 새로운 정보를 더해주지 않습니다.
C3Context token wasteR—context.token_waste답에 기여하지 않는 컨텍스트가 소비하는 토큰 수. 비용 낭비와 혼란을 동시에 유발합니다.
C4Evidence sufficiencyJL3context.insufficient조립된 컨텍스트가 완전한 정답을 만들기에 충분한 근거를 담고 있는가? 부족하면 응답이 불완전·부정확해집니다.
C5Context ordering qualityJ—context.order_bad청크가 논리적으로(가장 관련 있는 것이 먼저, 일관된 흐름으로) 정렬되어 있는가? 순서가 나쁘면 LLM이 정신적으로 재정렬해야 하므로 정확도가 떨어집니다.
C7Chunk semantic similarityR—context.sim_low청크가 원 질의와 얼마나 의미적으로 유사한가? 유사도가 낮다면 검색이 사용자 의도와 맞지 않는 청크를 통과시킨 것입니다.

Group D — Response Quality (12개 메트릭)

목적: 가장 큰 그룹입니다. 최종 생성된 응답을 관련성·정확성·근거성·형식·톤·안전성 등 여러 차원에서 평가합니다. 사용자가 직접 체감하는 영역입니다.

얻을 수 있는 인사이트: 답변에서 정확히 무엇이 잘못되었는지(환각? 불완전? 형식 오류? 안전성 위반?)를 짚어내며, 각 메트릭은 서로 다른 개선 전략과 연결됩니다.

CodeMetricModeGTCause Code설명 & 의의
D1Answer relevanceJ—gen.relevance_low답변이 사용자가 실제로 물은 것을 다루는가? 사실이 맞아도 무관한 답변은 사용자를 실망시킵니다.
D2CorrectnessJ/HL3gen.incorrect그라운드 트루스 대비 사실 내용이 정확한가? 가장 핵심적인 품질 차원입니다.
D3Faithfulness / groundednessJ—gen.unfaithful답변이 제공된 컨텍스트에 의해 뒷받침되는 주장만 포함하는가? 충실하지 않은 답변은 모델이 "지어낸" 정보를 끼워 넣습니다.
D4Hallucination rateJ—gen.hallucination답변 안의 주장 중 컨텍스트나 사실에 근거가 없는 비율. 5%만 되어도 사용자 신뢰가 무너질 수 있습니다.
D5CompletenessJL3gen.incomplete사용자가 물어본 모든 측면을 다루는가? 부분 답변은 추가 질문과 불만으로 이어집니다.
D6ConcisenessR/J—gen.verbose불필요한 군더더기 없이 적절히 간결한가? 장황한 답변은 사용자 시간과 토큰 비용을 동시에 낭비합니다.
D7Tone consistencyJ—gen.tone_off기대되는 톤(전문적·친근·기술적 등)이 일관되는가? 톤 불일치는 사용자에게 부자연스럽게 느껴집니다.
D8Policy complianceR/J—gen.policy_violation조직 정책(PII 금지, 경쟁사 언급 금지, 브랜드 가이드 등)을 준수하는가? 위반은 법적·브랜드 리스크로 이어집니다.
D9Citation accuracyRL3gen.citation_wrong출처를 인용할 때 실제로 정확한가? 잘못된 인용은 사용자를 호도하고 신뢰를 떨어뜨립니다.
D10Structured format validityR—gen.format_invalidJSON/XML 등 구조화 출력이 기대된다면 유효한가? 형식이 깨지면 출력을 파싱하는 후속 시스템이 실패합니다.
D11Language qualityJ—gen.grammar응답이 문법적으로 맞고 잘 쓰인가? 언어 품질이 낮으면 모델 역량 부족이나 프롬프트 문제일 가능성이 큽니다.
D12Refusal appropriatenessJ—gen.refusal_bad거절이 정당한가? 거짓 거절은 사용자를 좌절시키고, 거절 누락은 안전 리스크를 만듭니다.

Group E — Agent & Tool Quality (8개 메트릭)

목적: 에이전트의 오케스트레이션 계층 — 도구 선택, 인자 형식, 계획 수립, 멀티스텝 추론 — 을 평가합니다. RAG가 아닌 에이전트형 워크로드에서 특히 중요합니다.

얻을 수 있는 인사이트: 단순 파라미터 튜닝이 아니라 에이전트 아키텍처, 도구 설명, 계획 로직을 재설계해야 하는지 판단할 수 있습니다.

CodeMetricModeGTCause Code설명 & 의의
E1Tool selection accuracyR/JL1tool.wrong에이전트가 작업에 맞는 도구를 골랐는가? 도구 선택이 잘못되면 다른 요인이 아무리 좋아도 작업이 성공할 수 없습니다.
E2Tool argument validityR—tool.arg_invalid도구에 전달된 인자가 올바른 형식인가? 잘못된 인자는 도구 실행 실패와 재시도를 유발합니다.
E3Tool call success rateR—tool.fail도구 호출 중 성공한 비율. 낮으면 인증·스키마·가용성 등 시스템적 문제일 가능성이 큽니다.
E4Tool retry countR—tool.retry_high성공 전 재시도 횟수. 과도한 재시도는 지연과 비용을 증가시키며 보통 에러 처리가 부실하다는 신호입니다.
E5Unnecessary tool call ratioJ—tool.over_call필요 없는 도구를 호출했는가? 불필요한 호출은 시간과 비용을 낭비하고 부수 효과를 초래할 수 있습니다.
E6Multi-step plan correctnessJL1agent.plan_wrong멀티스텝 작업에서 에이전트가 유효한 계획을 만들었는가? 잘못된 계획은 단계 전반에 걸친 연쇄 실패로 이어집니다.
E7Reasoning path lengthR—agent.path_long에이전트가 거친 단계 수. 과도한 단계는 계획 루프나 결정 지연을 의미하며 토큰과 시간을 낭비합니다.
E8Final synthesis qualityJ—agent.synthesis_low여러 도구에서 정보를 모은 후 일관된 최종 답변을 만들었는가? 합성 품질이 낮으면 좋은 도구 결과도 헛수고가 됩니다.

Group F — Performance & Cost (5개 메트릭)

목적: 운영 건강 지표 — 지연 시간, 비용, 토큰 사용량, 신뢰성. 정확도가 완벽해도 너무 느리거나 비싸면 실패한 에이전트입니다.

얻을 수 있는 인사이트: 성능 이슈가 검색·생성·도구 호출·아키텍처 중 어디에 있는지 알 수 있고, 비용 거버넌스 의사결정에 직접 활용할 수 있습니다.

CodeMetricModeGTCause Code설명 & 의의
F1End-to-end latencyR—ops.latency_over사용자 질의에서 최종 응답까지의 총 시간. 사용자는 3~5초가 지나면 이탈하므로 만족도에 직접 영향을 줍니다.
F2Model inference latencyR—ops.gen_slowLLM 추론에만 사용된 시간. 모델 자체가 느린지, 다른 파이프라인 단계가 느린지 분리해서 볼 수 있습니다.
F3Cost per traceR—ops.cost_over한 번의 에이전트 호출당 USD 비용. 예산 준수 여부와 비용 비효율적인 트레이스를 식별합니다.
F4Token consumptionR—ops.token_over사용된 입력 + 출력 토큰 합계. 과도하다면 보통 장황한 프롬프트나 불필요한 컨텍스트 채움이 원인입니다.
F5Failure / timeout rateR—ops.failure에러 또는 timeout으로 끝난 트레이스 비율. 가장 기본적인 SLA 지표로서 신뢰성을 직접적으로 보여줍니다.

Group G — Human Outcomes (5개 메트릭)

목적: 자동 메트릭이 완전히 대체할 수 없는 실제 사용자 만족 신호를 포착합니다. 주관적 품질에서는 여전히 사람의 판단이 황금 기준입니다.

얻을 수 있는 인사이트: A~F의 자동 메트릭이 실제 사용자 만족과 얼마나 상관관계가 있는지 확인하고, 평가 프레임워크의 사각지대를 찾을 수 있습니다.

CodeMetricModeGTCause Code설명 & 의의
G1User thumbs up ratioH—human.dislike사용자의 명시적 긍정 피드백 비율. 가장 단순한 만족 신호로, 비율이 낮다면 사용자가 만족하지 않는다는 뜻입니다.
G2User dissatisfaction rateH—human.complaint명시적 부정 신호(불만, 에스컬레이션) 비율. 적극적인 불만을 보여주므로 thumbs-up보다 액셔너블합니다.

파이프라인 단계 흐름

기본 경로: A → B → C → D. E(에이전트 & 도구)는 보통 B~D와 병렬로 진행됩니다. F는 모든 단계에 걸쳐 측정되고, G는 응답 이후에 수집됩니다.

메트릭이 Improvement Pack과 연결되는 방식

메트릭이 임계값 아래로 떨어지면 시스템은 다음 절차를 수행합니다.

  1. cause code를 기록합니다 (예: retrieval.recall_low).
  2. cause code를 주 개선 카테고리(예: "Retrieval tuning")에 매핑합니다.
  3. 구체적 액션을 담은 Improvement Pack 제안을 생성합니다.
  4. 카테고리에 따라 노력 수준(Low/Medium/High)을 부여합니다.

이 폐쇄 루프 구조 덕분에, 모든 실패 메트릭이 구체적이고 우선순위가 매겨진 액션으로 자연스럽게 이어집니다.
