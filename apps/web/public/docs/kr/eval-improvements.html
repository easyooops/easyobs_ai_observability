Improvement Packs — 전체 기능 매뉴얼

Improvement Pack은 평가 실패를 실행 가능한 엔지니어링 작업으로 변환합니다. 각 팩은 품질 이슈의 근본 원인을 식별하고, 카테고리화된 해결 제안과 노력 등급을 제공합니다.

Improvement Pack 동작 방식

  1. 평가 실행이 reasoning과 함께 FAIL 또는 WARN verdict를 만듭니다.
  2. 시스템이 reasoning을 분석해 cause code로 매핑합니다.
  3. 각 cause code는 1차 카테고리와 2차 디테일에 매핑됩니다.
  4. 카테고리는 11개 그룹 중 하나에 속합니다.
  5. 각 제안은 노력 수준(Low / Medium / High)을 가집니다.
  6. 팩은 팀이 워크플로 상태를 통해 추적할 수 있는 액셔너블 카드 형태로 제공됩니다.

노력 수준

LevelBadge예상 시간예시
LowGreen< 1시간프롬프트 문구 조정, 가드레일 규칙 추가, temperature 조정
MediumYellow1시간 ~ 1일프롬프트 템플릿 재구성, retrieval 단계 추가, 청크 전략 개선
HighRed> 1일에이전트 아키텍처 재설계, 모델 fine-tune, 새로운 데이터 파이프라인 구축

워크플로 상태

각 improvement pack은 다음 상태들을 거칩니다.

State의미액션
open새로 식별되었으며 트리아지 대기 중.리뷰 후 우선순위 결정.
in_progress팀이 적극적으로 수정 작업 중.제안 사항 구현.
resolved수정 적용 완료, 검증 대기.평가를 다시 실행하여 확인.
dismissed해당 없음 또는 수정하지 않기로 결정.아카이브(히스토리에는 남음).

Cause Code 매핑

시스템은 LLM 저지의 reasoning 분석으로 생성된 52개의 내부 cause code를 사용하며, 이를 위 46개 카테고리로 매핑합니다. 여러 cause code가 같은 카테고리로 매핑될 수 있고, 이 매핑 덕분에 미묘한 실패 사유도 모두 액셔너블한 카테고리로 수렴됩니다.

UI/UX 흐름: 원시 judge finding이 안정된 카테고리로 정규화된 후, 노력·워크플로 컨트롤이 포함된 트리아지 카드로 렌더링됩니다.

UI 레이아웃

그룹화된 카드 뷰

  • improvement는 11개 카테고리 그룹으로 묶인 카드로 표시됩니다.
  • 그룹 헤더에는 그룹명과 open 항목 수가 표시됩니다.
  • 카드는 카테고리명, 노력 배지, 원인 요약, 제안 액션을 보여줍니다.
  • 카드를 클릭하면 상세를 펼쳐 보고 워크플로 상태를 변경할 수 있습니다.

필터

  • Effort: Low / Medium / High로 필터링.
  • State: Open / In Progress / Resolved / Dismissed.
  • Group: 특정 카테고리 그룹으로 필터링.
  • Run: 출처 평가 실행으로 필터링.

카드 상세 (펼친 상태)

필드설명
Category(46개 중) 1차 카테고리명.
Secondary Detail구체적인 상황 설명.
Group(11개 중) 상위 그룹.
Effort색상 배지가 있는 Low / Medium / High.
Cause Code추적성을 위한 내부 코드.
Source Run이 항목을 만든 평가 실행 링크.
Affected Traces이슈가 나타난 trace ID 목록.
Suggested Action시스템이 생성한 구체적인 다음 단계.
State Control버튼: Start Working / Resolve / Dismiss.

Runs와의 연동

improvement를 해결한 뒤 권장되는 절차는 다음과 같습니다.

  1. 에이전트 코드에 수정 사항을 적용합니다.
  2. 같은 트레이스에 대해 평가를 다시 실행합니다.
  3. FAIL/WARN verdict가 PASS로 전환되었는지 확인합니다.
  4. 검증이 완료되면 improvement를 "resolved"로 표시합니다.

권한

  • SA / PO: 모두 조회, 워크플로 상태 변경 가능.
  • DV: 모두 조회 (읽기 전용 — 상태 변경 불가).
