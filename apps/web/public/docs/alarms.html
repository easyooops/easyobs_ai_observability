Alarms — Detailed Manual

Alarms — Full Feature Manual

Alarms provide threshold-based alerting for key observability metrics. When a metric crosses a defined threshold within an evaluation window, the alarm transitions to a FIRING state and is prominently displayed on the dashboard.

Concept

An alarm continuously monitors a single metric and compares its value against a threshold:

IF metric_value {operator} threshold FOR {evaluation_window}
  THEN alarm = FIRING
ELSE
  alarm = OK

Alarm States

StateColorMeaning
OKGreenMetric is within acceptable range
FIRINGRedMetric has breached the threshold
PENDINGYellowThreshold breached but evaluation window not yet elapsed
DISABLEDGrayAlarm manually disabled by user
NO_DATAGrayInsufficient data to evaluate (e.g., no traces in window)

Available Metrics

MetricUnitAggregationDescription
error_rate%PercentageRatio of ERROR traces to total traces

trace_countcountSumTotal traces ingested in the window

Operators

OperatorSymbolMeaning
Greater than>Alert when metric exceeds threshold
Greater or equal>=Alert when metric is at or above threshold
Less than<Alert when metric drops below threshold
Less or equal<=Alert when metric is at or below threshold

Evaluation Window

The window determines how much recent data is considered:

WindowChecks EveryUse Case
5 minutes~1 minRapid detection for high-traffic services
15 minutes~5 minStandard monitoring
1 hour~10 minSmoothed alerting, less noisy
6 hours~30 minTrend detection over longer periods
24 hours~1 hrDaily budget/volume thresholds

Creating an Alarm

  1. Navigate to Setup → Alarms
  2. Click "Create Alarm"
  3. Fill in the form:
FieldRequiredDescription
NameYesDescriptive name (e.g., "High Error Rate Alert")
MetricYesSelect from available metrics dropdown
OperatorYesComparison operator
ThresholdYesNumeric value to compare against
Evaluation WindowYesTime range to aggregate metric over
Service FilterNoScope to specific service (null = all services)
DescriptionNoAdditional context for the team
EnabledYesWhether alarm is active
  1. Click Save
  2. Alarm immediately begins evaluation

Alarm Examples

Example 1: High Error Rate

Name: "Error Rate > 5%"
Metric: error_rate
Operator: >
Threshold: 5.0
Window: 15 minutes
Service: all

Fires when more than 5% of traces in the last 15 minutes have ERROR status.

Example 2: Latency Spike

Name: "P95 Latency > 3s"
Metric: p95_latency
Operator: >
Threshold: 3000
Window: 5 minutes
Service: "customer-support-agent"

Fires when 95th percentile latency exceeds 3 seconds for the specific service.

Example 3: Cost Budget

Name: "Daily cost > $50"
Metric: total_cost
Operator: >
Threshold: 50.0
Window: 24 hours
Service: all

Fires when total LLM spend in a 24-hour rolling window exceeds $50.

Example 4: Quality Degradation

Name: "Pass rate below 80%"
Metric: eval_pass_rate
Operator: <
Threshold: 80.0
Window: 6 hours
Service: all

Fires when the evaluation pass rate drops below 80%.

Dashboard Integration

  • FIRING alarms are pinned to the Overview dashboard as red cards
  • Each pinned card shows: alarm name, current value vs. threshold, time in FIRING state
  • Cards disappear when alarm returns to OK
  • Up to 5 alarms are pinned; if more are firing, a "+N more" link appears

State Transitions

UI view: alerts move through clear visual states so operators can decide whether to wait, investigate, or acknowledge.

  • OK → PENDING: Metric breaches threshold (first detection)
  • PENDING → FIRING: Metric still breaching after full evaluation window
  • FIRING → OK: Metric returns within threshold
  • PENDING → OK: Metric recovers before window elapses (no false alarm)

Editing & Deleting Alarms

  • Edit: Click alarm row → modify any field → Save
  • Disable: Toggle the Enabled switch → alarm stops evaluating
  • Delete: Click delete icon → confirmation → permanently removes
  • Disabled alarms retain their configuration and history

Permissions

  • SA / PO: Create, edit, delete, enable/disable alarms
  • DV: View alarm list and current states (read-only)

Best Practices

  • Start with conservative thresholds, then tighten as you understand your baseline
  • Use the 15-minute window as default — balances responsiveness vs. noise
  • Create separate alarms per critical service rather than one global alarm
  • Combine metric alarms with quality alarms for comprehensive coverage
  • Review and adjust thresholds quarterly as traffic patterns change
