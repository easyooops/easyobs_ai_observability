Evaluation Profiles — Detailed Manual

Evaluation Profiles — Full Feature Manual

An Evaluation Profile is a reusable configuration template that defines what to evaluate, how to evaluate, and how much to spend. Profiles are selected when launching evaluation runs.

Profile Structure

FieldRequiredDescription
NameYesHuman-readable profile name (e.g., "Production Safety Check")
DescriptionNoOptional description of the profile's purpose
Project / ServiceNoScope to a specific service (limits visible traces). Null = all services
Evaluator TypesYesWhich evaluator types to include (Rule / LLM Judge / Human)
DimensionsYesWhat aspects to evaluate (list of dimension configs)
Consensus PolicyYesHow multi-judge results are combined
Budget GuardYesMaximum USD cost allowed per run
ActiveYesWhether this profile appears in the run launcher

Evaluator Types

A profile can include one or more evaluator types:

TypeHow It WorksCost
Rule-basedDeterministic checks (regex, keyword, length, JSON validity)Free (server-side only)
LLM JudgeSends trace to registered judge models for assessmentPer-token (depends on judge model)
HumanRoutes to human reviewers for manual labelingFree (but requires human time)

Evaluation Dimensions (52 Metric Catalog)

Complete Reference: EasyObs ships with a 52-metric catalog spanning 7 groups (A–G). Each metric has a specific cause code that drives Improvement Packs. View the full 52 Metric Catalog with detailed explanations

Dimensions define the specific aspects being evaluated. Common dimension examples:

DimensionWhat It AssessesTypical Criteria
HelpfulnessDid the response adequately address the user's need?Completeness, relevance, actionability
SafetyIs the response free from harmful content?No hate speech, no PII, no dangerous instructions
AccuracyIs the factual content correct?Verified against sources, no hallucination
CoherenceIs the response logically structured?No contradictions, clear flow, on-topic
Instruction FollowingDid the agent follow the system prompt?Format compliance, scope adherence
Cost EfficiencyWas the response produced efficiently?Reasonable token usage, no redundant calls

Each dimension produces its own score (0.0–1.0) and verdict. The overall verdict is the aggregate across all dimensions.

Consensus Policy

When multiple LLM judges are used, the consensus policy determines the final verdict:

PolicyLogicBest For
singleUses only the first enabled judge's verdictSpeed, low cost
majority>50% of judges must agree on the verdictBalanced reliability
unanimousAll judges must agree, else verdict = WARNHigh-stakes, conservative
weightedWeighted average using each judge's weight fieldExpert ensemble

Budget Guard

The budget guard prevents runaway evaluation costs:

  • Per-run limit: Maximum USD that a single run can spend on judge API calls
  • Estimation: Before launch, the system estimates cost and blocks if it exceeds the guard
  • Formula: estimated_cost = subject_count × judges_count × avg_tokens_per_eval × token_rate
  • Runtime enforcement: If a running evaluation approaches the limit, it stops early and marks remaining subjects as "budget_exceeded"

Creating a Profile

  1. Navigate to Quality → Profiles
  2. Click "Create Profile"
  3. Fill in the form:
    • Name and description
    • Select project/service scope (optional)
    • Choose evaluator types
    • Define dimensions with criteria text
    • Set consensus policy
    • Set budget guard (USD)
  4. Click Save

Editing & Cloning

  • Edit: Click profile row → modify fields → Save
  • Clone: Create a copy of an existing profile with a new name (useful for A/B testing evaluation criteria)
  • Deactivate: Set active=false to hide from the run launcher without deleting

API Access

# List profiles
GET /api/eval/profiles

# Create profile
POST /api/eval/profiles
{
  "name": "Safety Eval v2",
  "projectId": "service-abc",
  "evaluatorTypes": ["rule", "llm_judge"],
  "dimensions": [...],
  "consensusPolicy": "majority",
  "budgetGuardUsd": 5.00,
  "active": true
}

Permissions

  • SA / PO: Full CRUD on profiles
  • DV: Read-only (can view profiles but not create/edit/delete)

Best Practices

  • Create separate profiles for different quality aspects (safety vs. helpfulness vs. accuracy)
  • Start with a low budget guard ($1–5) during experimentation
  • Use "majority" consensus as the default for production evaluations
  • Scope profiles to specific services to avoid evaluating irrelevant traces
  • Keep descriptions detailed — they help team members understand the profile's intent
  • For fine-grained prompt control per dimension, use the Evaluation Prompts tab under Judges — profile-level overrides take priority over versioned prompts
