LLM Judges ??Full Feature Manual

LLM Judges are external language model endpoints registered in EasyObs to serve as automated evaluators. They assess trace quality by analyzing LLM inputs/outputs and producing verdicts with reasoning.

?“¸ Image: Judges page ??table of registered judges with provider badges

Concept

An LLM Judge is a configured model endpoint that:

  1. Receives a structured evaluation prompt (trace context + evaluation criteria)
  2. Returns a verdict (PASS / WARN / FAIL), a numeric score (0.0??.0), and reasoning text
  3. Can be combined with other judges via consensus policies for more reliable results

Multi-Judge Consensus

When an evaluation profile uses multiple judges, their verdicts are combined using a consensus policy (configured per profile):

PolicyLogicUse Case
SingleFirst judge's verdict is finalFast, cost-efficient evaluations
Majority>50% of judges must agreeBalanced reliability vs. cost
UnanimousAll judges must agree, else WARNHigh-stakes decisions where any doubt = flag
WeightedVerdicts weighted by judge's weight fieldExpert ensemble (e.g., specialized judge gets 3x weight)

Judge Weight in Weighted Consensus

Example: 3 judges with weights 1.0, 1.0, 3.0 (total = 5.0)

  • Judge A (w=1.0): PASS ??contributes 1.0 to PASS
  • Judge B (w=1.0): FAIL ??contributes 1.0 to FAIL
  • Judge C (w=3.0): PASS ??contributes 3.0 to PASS
  • Result: PASS score = 4.0/5.0 = 0.80 ??verdict PASS

Editing & Deleting

  • Edit: Click any row ??form opens with current values ??modify ??Save
  • Delete: Click trash icon ??confirmation dialog ??permanently removes the judge
  • Deleting a judge that is referenced by active profiles will not break existing runs but will prevent new runs from using it

Permissions

  • SA / PO: Full CRUD (create, read, update, delete)
  • DV: Read-only view of judges table

Best Practices

  • Register at least 2 judges from different providers for cross-validation
  • Use temperature: 0 for maximum reproducibility
  • Set accurate cost values — they directly affect budget guard calculations
  • Name judges descriptively (e.g., "GPT-4o Strict Safety" vs. "Claude Helpfulness")
  • Disable judges you're experimenting with to exclude them from production runs

Evaluation Prompts (Versioned Prompt Management)

The Evaluation Prompts tab (next to Judge Models) allows organizations to customize the system and user message prompts used by the LLM-as-a-Judge for each evaluation dimension.

Concept

Each evaluation dimension (e.g., Faithfulness, Answer Relevance, Coherence) has its own prompt pair:

  • System Prompt — instructions to the judge LLM about how to evaluate
  • User Message Template — the structured payload sent with placeholders ({rubric_id}, {rubric}, {context_json})

Prompts are versioned: v1 (initial/default) → v2 → v3 … Every edit creates a new version. Only the active version is used during evaluation. Previous versions are preserved for auditing and rollback.

How to Use

  1. Navigate to Quality → Judges → Evaluation Prompts tab
  2. Select a dimension from the left panel
  3. View existing versions (or see "default" if no custom prompt has been created)
  4. Click "New version" to create a customized prompt
  5. Edit the System Prompt and/or User Message Template
  6. Click "Save as new version"

Version Management

ActionBehavior
Create first promptSaved as v1, automatically set as active
Edit / new versionPrevious version deactivated, new version becomes active
Activate older versionSwitches the active flag (rollback without deleting)
Clone & EditPre-fills editor with an existing version's content

How Prompts Flow into Evaluation

Priority order when the judge LLM is called:

  1. Profile-level override — if the evaluation profile has judge_system_prompt or judge_user_message_template set, those take highest priority
  2. DB versioned prompt (active) — if no profile override, the active versioned prompt for the dimension is used
  3. Built-in default — if neither exists, the system falls back to the hardcoded defaults
