52 Evaluation Metrics ??Complete Catalog

EasyObs evaluates agent quality using a stage-aligned metric framework covering the full pipeline: Query ??Retrieval ??Context ??Response ??Agent/Tool ??Operations ??Human Feedback. Each metric answers a specific "what went wrong?" question and maps to actionable improvement categories.

How to Read This Catalog

ColumnMeaning
CodeShort identifier (e.g., A1, B3, D4)
ModeR = Rule (deterministic, free), J = LLM Judge (token-priced), H = Human label
GTGolden-set layer required: L1 (intent), L2 (retrieval docs), L3 (expected output), ??= none needed
Cause CodeStructured failure code attached to findings ??drives Improvement Packs
Why 52 metrics? Agents fail for different reasons at different pipeline stages. A single "quality score" cannot tell you what to fix. By decomposing quality into 52 specific metrics across 7 groups, EasyObs pinpoints the exact failure stage and maps it to concrete remediation actions via Improvement Packs.

Group A ??Query Understanding (5 metrics)

Purpose: Evaluates whether the agent correctly understood the user's intent before any processing begins. Failures here cascade to all downstream stages.

What you learn: If your agent misclassifies intent, rewrites queries incorrectly, or fails to handle ambiguous/multilingual inputs ??fix these first before tuning retrieval or generation.

A1Intent classification accuracyR/JL1query.intent_mismatchDoes the agent identify the correct intent (e.g., "order status" vs. "returns")? Misclassification routes the query to wrong workflows, causing complete failure.
A2Query rewrite semantic preservationJ??/td>query.rewrite_driftWhen the agent rewrites/expands the query for search, does it preserve the original meaning? Drift causes irrelevant retrieval results.
A3Ambiguity detection rateJ??/td>query.ambiguousCan the agent detect when a query is too vague and needs clarification? Missing this causes hallucinated or random answers.
A5Language detection accuracyR??/td>query.lang_mismatchDoes the agent respond in the same language as the user? Language mismatch destroys user experience.
A6Query complexity scoreR??/td>query.complexity_highFlags queries that exceed the agent's capability ceiling. Helps route complex queries to human agents or specialized models.

Group B ??Retrieval Quality (10 metrics)

Purpose: Measures whether the RAG retrieval stage finds the right documents with good ranking. This is the most metric-dense group because retrieval is the #1 failure point in RAG systems.

What you learn: Exactly which retrieval sub-system needs fixing ??embedding model, chunking strategy, reranker, index coverage, or latency.

CodeMetricModeGTCause CodeWhat It Measures & Why
B1Recall@KRL2retrieval.recall_lowOf all relevant documents, what fraction did the system retrieve in the top K? Low recall = missing critical information that should be in the answer.
B2Precision@KRL2retrieval.noise_highOf the K documents retrieved, what fraction is actually relevant? Low precision = context polluted with noise, confusing the LLM.
B3MRR (Mean Reciprocal Rank)RL2retrieval.first_hit_lateHow early does the first relevant document appear? Low MRR means the model must scan deep into context to find useful information.
B4HitRate@KRL2retrieval.missBinary: did at least one relevant document appear in top K? A "miss" means zero relevant context was retrieved ??guaranteed bad answer.
B5nDCG@KRL2retrieval.rank_qualityNormalized Discounted Cumulative Gain ??evaluates ranking quality weighted by position. Relevant docs ranked lower contribute less.
B6MAP (Mean Average Precision)RL2retrieval.precision_lowAverage precision across all relevant docs ??penalizes when relevant docs are scattered among irrelevant ones rather than clustered at top.
B7Context precision (judge)J??/td>retrieval.chunk_irrelevantLLM judge assesses: are the retrieved chunks actually useful for answering this query? Catches semantic irrelevance that keyword metrics miss.
B8Context recall (judge)JL3retrieval.coverage_lowLLM judge assesses: does the retrieved context contain enough information to produce the expected answer? Detects knowledge gaps.
B10Reranker gainRL2retrieval.rerank_no_gainDoes the reranker improve ranking vs. raw retrieval? Zero gain means the reranker adds latency without benefit ??consider removing it.
B11Retrieval diversityR??/td>retrieval.dup_highAre retrieved chunks too similar to each other? High duplication wastes context window on redundant information.
B12Retrieval latency budgetR??/td>retrieval.latency_overDoes retrieval complete within the time budget? Slow retrieval delays the entire response and degrades UX.

Group C ??Context Quality (6 metrics)

Purpose: Evaluates the quality of context after retrieval but before generation. Even if retrieval finds good documents, poor context assembly (chunking, ordering, deduplication) degrades LLM output.

What you learn: Whether your context assembly pipeline needs improvement ??chunk sizes, ordering strategy, deduplication, or semantic similarity filtering.

CodeMetricModeGTCause CodeWhat It Measures & Why
C1Chunk noise ratioR/J??/td>context.noiseWhat fraction of context tokens are irrelevant to the query? High noise dilutes the signal, causing the LLM to lose focus or hallucinate.
C2Duplicate chunk ratioR??/td>context.dupWhat fraction of chunks are near-duplicates? Duplicates waste context window without adding new information.
C3Context token wasteR??/td>context.token_wasteHow many tokens are consumed by context that doesn't contribute to the answer? Direct cost waste + potential confusion.
C4Evidence sufficiencyJL3context.insufficientDoes the assembled context contain enough evidence to produce a complete, correct answer? Insufficiency guarantees incomplete/wrong responses.
C5Context ordering qualityJ??/td>context.order_badAre chunks ordered logically (most relevant first, coherent flow)? Bad ordering forces the LLM to mentally reorder, reducing accuracy.
C7Chunk semantic similarityR??/td>context.sim_lowHow semantically similar are chunks to the original query? Low similarity = retrieval passed chunks that don't match user intent.

How Metrics Connect to Improvement Packs

When a metric fails (score below threshold), the system:

  1. Records the cause code (e.g., retrieval.recall_low)
  2. Maps the cause code to primary remediation category (e.g., "Retrieval tuning")
  3. Generates an Improvement Pack proposal with specific actions
  4. Assigns an effort level (Low/Medium/High) based on the category

This closed-loop system ensures every failed metric directly leads to a concrete, prioritized action item.
