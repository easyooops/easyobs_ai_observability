Evaluation Runs ??Full Feature Manual

The Runs page is the primary workbench for launching, monitoring, and reviewing automated quality evaluations. It combines a 3-way tabbed interface with 4 source strategies, cost estimation, and detailed result drill-down.

Page Structure ??3-Way Workbench

The top of the Runs page has a segmented control with 3 tabs:

TabURL ParamPurpose
Launch?tab=launch (default)Configure and fire new evaluation runs
Results?tab=resultsBrowse completed runs, inspect verdicts and scores
Background Hub?tab=bgMonitor in-progress runs with live status

Tab 1: Launch

The Launch tab is a step-by-step wizard for configuring and firing an evaluation run.

Step 1 ??Select Evaluation Profile

A dropdown lists all active evaluation profiles (fetched from GET /api/eval/profiles). Each profile defines:

  • Which evaluator types to use (Rule / LLM Judge / Human)
  • Consensus policy (single, majority, unanimous, weighted)
  • Budget guard (max cost per run)
  • Target project/service scope
Tip: Create profiles in Quality ??Profiles before launching your first run.

Step 2 ??Choose Source (4 Source Cards)

Four source cards determine which traces are evaluated:

#SourceIDRun ModeDescription
1 Time Window window trace All traces within the current time window (1h/6h/24h/7d). Pick individual traces from the list or "Select All".
2 Single Trace single_trace trace Paste or pre-fill a specific trace ID. Entry point from the Trace Inspector "Evaluate ?? button.
3 Session session trace Filter traces by session ID ??evaluates an entire conversation thread.
4 Human Label Cohort human_label golden_judge Uses previously human-labeled annotations as ground truth. Compares LLM judge verdicts against human verdicts.
?“¸ Image: 4 Source Cards ??horizontal layout with active highlight

Source: Time Window (detail)

  • Fetches up to 2,000 traces matching the profile's project scope
  • Groups traces by session for easy browsing
  • Checkbox per trace ??toggle individually or "Select All in View"
  • Collapsible session groups (click session header to expand/collapse)
  • Shows trace operation name, service, duration, status badge, cost

Source: Single Trace (detail)

  • Text input field for a 32-char trace ID
  • Auto-filled via ?prefill-trace=<id> URL param (from Trace Inspector CTA)
  • Validates the trace exists before allowing launch

Source: Session (detail)

  • Text filter for session ID (substring match)
  • Displays all matching traces ??select the ones you want evaluated
  • Useful for evaluating multi-turn conversation quality

Source: Human Label Cohort (detail)

  • Lists all human label annotations (up to 200 recent)
  • Filter by verdict: All / Pass / Warn / Fail
  • Multi-select individual annotations
  • Run mode switches to golden_judge ??compares LLM judge output vs. human labels
  • Falls back to golden_gt mode if no judge models are registered

Step 3 ??Cost Estimation

Before launching, click "Estimate Cost" to preview:

MetricDescription
Estimated CallsNumber of LLM judge API calls required
Estimated CostProjected USD cost based on judge model pricing
Budget RemainingHow much of the profile's budget guard remains
Trace CountNumber of selected subjects
Warning: If estimated cost exceeds the profile's budget guard, the Launch button is disabled with an explanation.

Step 4 ??Notes & Launch

  • Notes (optional) ??Free text field for context (e.g., "Testing new prompt v2")
  • Launch Run ??Fires the evaluation. The UI switches to Results tab with the new run selected.

Run detail ??full workbench

From the Results tab, click any run row to open the run workbench. The same view is used when you land with ?tab=results&run=<id>. While a run is still running, the UI polls for updates about every 4 seconds.

Layout

A two-column layout: a sticky left sidebar (always visible while scrolling) and a main panel whose title shows the active tab name, run mode (e.g. trace, golden_judge), and optional golden set id when applicable.

Sticky sidebar ??run context

  • Run id ??Short prefix of the run UUID and a status badge (succeeded / failed / other).
  • cost ??Actual USD vs estimated USD (costActualUsd / costEstimateUsd).
  • trust ??A 1?? star summary derived from the five trust gauges (mean of available gauge scores).
  • source ??Inferred label: Golden Set (golden run modes), Human-labeled, Single Trace, or Window.
  • profile ??Evaluation profile id (short).
  • by ??Who triggered the run (or system).
  • started ??Relative time.
  • Tab buttons ??Switch the seven main tabs (see below).

Tab: Replay

When a parent run exists, this area compares the same subjects across runs (verdict drift, pass rate, average score, ?) via slope-style charts. If there is no parent_run_id, the UI explains that the run is standalone or the parent is not recorded yet.

Tab: Findings

Flattens every finding on every result, groups by evaluatorId, and lists trace prefix, verdict, score, and reason (first 12 per group, with overflow count).

Tab 3: Background Hub

Monitors all in-progress or recently completed runs with live polling (every 10 seconds).

  • Shows run status progression: queued ??running ??completed/failed
  • Progress bar with percentage for running tasks
  • Click any row to jump to the Results tab with that run selected
  • Cancelled runs are shown with a strikethrough style

Permissions

  • Project Owner (PO) and Super Admin (SA) ??Full access: launch, view, cancel runs
  • Developer (DV) ??Read-only: can view results but cannot launch or cancel
  • If the user lacks write permission, a WriteHint banner appears and the Launch button is disabled

Run Lifecycle

User clicks "Launch Run"
  ??POST /api/eval/runs (creates run record, status=pending)
  ??Background worker picks up the task
      ??For each trace subject:
          1. Fetch trace data + LLM spans
          2. Execute each evaluator in the profile
          3. Apply consensus policy
          4. Record verdict + reasoning
      ??Update run status ??completed | failed
  ??UI auto-refreshes (10s interval)
