Sessions — Detailed Manual

Sessions — Full Feature Manual

Sessions group multiple traces (agent turns) into a single logical conversation thread. This enables end-to-end analysis of multi-turn interactions, from first user message to final resolution.

📸 Image: Sessions page — list of sessions with turn count, duration, and status

Concept

A session is a logical grouping of traces that share the same o.sess attribute. Typical use cases:

  • Multi-turn chatbot conversations (user asks follow-ups)
  • Multi-step agent workflows (agent iterates toward a goal)
  • A/B test cohorts (group traces by experiment ID)

How Sessions Are Created

Sessions are not explicitly created — they emerge automatically when traces share the same session ID. Set the session ID in your agent code:

Python SDK

from easyobs_agent import init, traced

init(session="conv-abc-123")

@traced
def handle_turn(user_msg: str):
    ...

OTLP Attributes

span.set_attribute("o.sess", "conv-abc-123")

LangChain Callback

from easyobs_agent import init
init(session="conv-abc-123", auto_langchain=True)

Session Detail View

Click any session row to open the detail view. It shows:

Timeline

  • Chronological list of all traces in the session
  • Each trace shows: operation name, duration, tokens, cost, status
  • Visual timeline bar showing relative timing of each turn

Conversation Replay

  • Displays o.q (user query) and o.r (agent response) for each trace
  • Formatted as a chat interface (user on left, agent on right)
  • Shows intermediate tool calls and retrieval results between turns

Navigation from Sessions

  • Click trace row → Opens Tracing page with ?session=<id> filter pre-applied
  • "Evaluate Session" button → Jumps to Quality > Runs with source=session and session ID pre-filled
  • "View Trace" button → Opens individual trace in the Trace Inspector

Filtering & Search

  • Time Window: Only shows sessions with at least one trace in the current time window
  • Search: Filters session list by session ID substring match
  • Service filter: Shows only sessions involving a specific service

Session Tagging

Additional context can be attached to traces within a session:

AttributePurposeExample
o.sessSession ID (required for grouping)"conv-abc-123"
o.userEnd-user identifier"user-456"
o.qUser query text"What's the weather?"
o.rAgent response text"It's 72°F and sunny."
o.turnTurn number (auto-incremented)3

Best Practices

  • Use consistent, unique session IDs (UUIDs recommended)
  • Set o.user to correlate sessions with end-users
  • Keep session IDs stable across reconnections (e.g., use a persistent conversation ID from your app)
  • For A/B tests, prefix session IDs: exp-v2-{uuid} for easy filtering
